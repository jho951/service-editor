# 2026-03-31 Troubleshooting Doc System

## Step 1. Learn 경로와 troubleshooting 전용 디렉토리 분리

- 목적: 개인 학습용 로컬 문서와 트러블슈팅 기록이 공식 구조 설명 문서 하위처럼 보이지 않도록 `docs/learn/`, `docs/learn/troubleshooting/` 구조로 분리한다.
- 변경 내용: 기존 학습 문서와 트러블슈팅 문서를 `docs/learn/` 아래로 이동하고, 관련 경로 참조와 ignore 경로를 함께 정리했다.
- 판단: 트러블슈팅은 공식 explainer보다 개인 학습과 회고 성격이 강하므로 별도 경로로 분리하는 편이 목적에 맞다.

## Step 2. troubleshooting 문서 위치와 README 기준 확정

- 목적: 문제 해결 과정을 어디에 어떤 기준으로 남길지 전역 규칙과 디렉토리 기준을 분리해 고정한다.
- 변경 내용: `AGENTS.md`에는 `docs/learn/troubleshooting/`에 기록하고 해당 경로의 `README.md` 기준을 따른다는 라우팅 규칙만 남겼다.
- 변경 내용: `docs/learn/troubleshooting/README.md`에 범위, 권장 흐름, 섹션별 작성법, 기본 템플릿과 체크리스트를 정리했다.
- 판단: 상위 가이드는 짧게 유지하고, 실제 작성 기준은 troubleshooting README에 집중시키는 구조가 더 명확하다.

## Step 3. 하나의 문제는 하나의 문서 원칙과 최소 완료 기준 추가

- 목적: 같은 문제 축의 기록이 여러 파일로 흩어지지 않고, 재현부터 해결까지 한 흐름으로 남게 한다.
- 변경 내용: 같은 문제 축의 트러블슈팅은 하나의 문서에 누적하고, 새 문서는 문제 축이 실제로 달라졌을 때만 만든다는 기준을 반영했다.
- 변경 내용: 문제 요약, 재현 조건, 원인 분석, 해결 후보 비교, 채택 이유, 결과와 검증이 없으면 충분하지 않다고 보는 최소 완료 기준을 추가했다.
- 변경 내용: 문서 마지막에 `문제 발생`, `문제 해결 과정`, `결론` 요약 블록을 두는 기준을 추가했다.

## Step 4. 의무 대상과 작성 인상 기준 보강

- 목적: 실제 장애 대응, 오류 해결, 성능 분석, 동시성 이슈 같은 작업에서 문서가 빠지지 않고, 나중에 읽어도 근거가 남는 기록이 되게 한다.
- 변경 내용: 오류 해결, 성능 분석, flaky test, 환경/배포 문제, 해결안 비교 후 채택 같은 경우를 의무 대상 예시로 정리했다.
- 변경 내용: 단순 개념 설명, 사소한 문구 수정, 기계적 rename 같은 비대상 예외도 함께 적었다.
- 변경 내용: 추상적 요약보다 관찰 사실, 버린 가설, 채택 이유, 테스트/로그/쿼리/코드 경로 같은 검증 가능한 흔적을 중심으로 쓰라는 기준과 예시를 추가했다.
