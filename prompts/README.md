# 프롬프트 카탈로그

구현, 리뷰, 장애 대응에 재사용할 작업 로그와 주제 탐색 문서를 정리한다.

## 디렉토리 구조

- [prompts/worklog/](https://github.com/jho951/Block-server/blob/dev/prompts/worklog/README.md)
  - 원본 작업 로그 보관
  - 월 단위 디렉토리(`YYYY-MM/`) 아래에 날짜별 로그 파일 유지

- [prompts/topics/](https://github.com/jho951/Block-server/blob/dev/prompts/topics/README.md)
  - 주제별 탐색 문서
  - 관련 worklog, REQUIREMENTS, discussion, ADR, explainer, guide 링크를 모아 다시 찾기 쉽게 정리

- `prompts/plans/`
  - 계획 템플릿과 중간 계획 문서
  - 보조 자료 성격이며 원본 작업 로그 보관 위치로 사용하지 않음

## 디렉토리 역할

- `worklog`는 사실 기록을 쌓는 곳이다. 작업 중 실제로 일어난 설계, 구현, 검증 흐름은 여기에 남긴다.
- `topics`는 찾기 위한 문서다. 같은 주제를 다시 볼 때 필요한 worklog와 공식 문서 링크를 짧게 묶는다.
- `topics`는 원본 로그를 복제하지 않는다. 요약과 링크만 유지하고, 세부 사실은 `worklog`를 기준으로 본다.

## 작업 로그 운영 원칙

- 작업 로그는 팀 또는 프로젝트 단위의 의미 있는 작업 기록이 필요할 때만 남긴다.
- 개인 학습용 설명 요청, 간단한 질의응답, 잠깐 확인하고 끝나는 요청은 로그를 남기지 않는다.
- 작업 로그 안의 내부 문서 링크는 로컬 절대 경로 대신 GitHub 저장소 기준 `blob/dev` 링크를 사용한다.
- 작업 로그는 가능한 한 목표 단위의 단일 파일로 유지한다.
- 작업 로그는 [prompts/worklog/](https://github.com/jho951/Block-server/blob/dev/prompts/worklog/README.md) 아래 `YYYY-MM/` 디렉토리에 둔다.
- 로그 파일명은 가능하면 `날짜 + 목표명` 형태의 고정 파일명으로 잡고, 같은 주제의 후속 이행과 검증도 해당 파일에 계속 누적한다.
- 같은 목표 아래 이어지는 작업은 새 파일을 계속 만들지 말고 기존 파일에 `Step`을 추가하는 방식으로 누적한다.
- 작업 목표는 사용자가 요청한 상위 흐름 기준으로 잡는다. 구현 중 파생된 세부 수정, 문서 보강, 후속 검증, 리팩터링은 같은 사용자 목표를 계속 수행하는 한 별도 파일로 분리하지 않는다.
- 커밋을 여러 개로 나누거나 중간에 문서 구조를 다시 정리하더라도, 하나의 궁극적 목표를 향한 연속 작업이면 같은 로그 파일에 계속 누적하는 것을 기본 원칙으로 한다.
- 새 로그 파일은 목표가 실질적으로 달라졌거나, 기존 로그가 지나치게 길어져 별도 트랙으로 분리하는 편이 더 명확할 때만 만든다.
- 각 단계는 `## Step n. 제목` 형식으로 작성한다.
- 각 Step 안에는 필요할 때 `목적`, `변경 내용`, `판단`, `검증`을 짧게 적어 흐름이 다시 읽히게 정리한다.
- 모든 세부 대화나 자잘한 정리까지 `Step`으로 남기지는 않고, 목표를 실질적으로 전진시키는 설계 결정, 구현 단계, 검증 결과 중심으로만 기록한다.

## Topic 운영 원칙

- `topics` 문서는 반복 조회 가치가 있는 주제에만 만든다.
- topic 문서는 원칙적으로 하나의 주제를 대표하는 단일 파일로 유지한다.
- topic 문서에는 아래 정도만 짧게 둔다.
  - 주제 목적
  - 관련 worklog 링크
  - 관련 REQUIREMENTS, discussion, ADR, explainer, guide 링크
  - 현재 기준 또는 최종 결론
  - 열어둘 질문
- topic 문서는 원본 worklog의 세부 내용을 다시 길게 복사하지 않는다.
- 관련 worklog가 추가되면 새 topic 파일을 만들기보다 기존 topic 문서에 링크를 보강한다.
