# Document Trash v2 Roadmap

## 현재 기준

- 기본 문서 삭제 API는 hard delete다.
- 휴지통 이동은 기본 삭제와 분리된 `PATCH /v1/documents/{documentId}/trash` 엔드포인트로 제공한다.
- 휴지통 상태는 별도 상태 필드 없이 `deletedAt is not null`로 표현한다.
- 휴지통 진입 시 대상 문서, 하위 문서, 각 문서 소속 블록을 함께 soft delete 처리한다.
- 휴지통 문서는 일반 문서 목록, 단건 조회, 콘텐츠 조회의 기본 결과에서 제외한다.
- 현재 테스트 기준 자동 삭제 시간은 `deletedAt + 5분`이다.
- 현재 테스트 기준 5분 이내에는 문서 복구를 허용하고, 부모 문서가 삭제 상태인 경우 자식 문서 단독 복구는 허용하지 않는다.
- 기본 문서 삭제 시 대상 문서, 하위 문서, 각 문서 소속 블록을 즉시 물리 삭제한다.
- 자동 영구 삭제 시 대상 문서, 하위 문서, 각 문서 소속 블록을 함께 물리 삭제한다.

## 후속 Todo

- 자동 삭제 시간을 설정값으로 분리해 운영 정책 변경 시 코드 수정 범위를 줄일지 검토
- 기본 삭제와 휴지통 이동을 클라이언트 UX에서 어떻게 구분 노출할지 검토
- 휴지통 목록 조회 API 제공 여부 검토
- 휴지통 복구 실패 시 만료 전용 오류 코드를 둘지 검토
- 자동 영구 삭제 실행 결과를 운영 로그나 메트릭으로 얼마나 남길지 정리

## 검증 보강

- `4분 59초` 경과 문서가 자동 삭제 대상에 포함되지 않는지 검증
- 기본 hard delete 요청이 하위 문서와 소속 블록까지 즉시 정리하는지 검증
- 자동 삭제 직전 복구 요청과 스케줄러 실행이 경합할 때 결과 일관성 검증
- 깊은 문서 트리와 많은 블록을 가진 문서에서 자동 삭제 쿼리 수와 실행 시간 측정
- 자동 삭제 대상이 없을 때 스케줄러가 안전 종료되는지 검증

## 정책 / 운영 고려사항

- 현재 5분은 테스트 기준 시간이며, 실제 운영 정책 확정 시 더 긴 보관 시간으로 변경될 수 있다.
- 기본 delete가 hard delete이므로 end-user 노출 시 확인 UX가 필요할 수 있다.
- 사용자 문구는 "현재 테스트 기준 5분 후 자동 삭제"처럼 구현과 다르게 과장되지 않게 맞춰야 한다.
- 자동 삭제가 최대 1분 이내 지연될 수 있는지, 더 정밀한 실행 보장이 필요한지 운영 기준을 정해야 한다.
- 문서와 하위 문서, 소속 블록을 같은 생명주기로 관리해야 dangling data가 남지 않는다.

## 선행조건 / 의존성

- 자동 삭제 실행 주체를 애플리케이션 스케줄러로 둘지 운영 배치로 둘지 최종 결정 필요
- 기본 hard delete를 실제 사용자 기능으로 열지, 운영/관리 경로로 제한할지 결정 필요
- 복구 가능 시간 변경 시 요구사항 문서와 테스트 데이터를 함께 조정해야 함
- 운영 환경에서 자동 삭제 로그 또는 모니터링 수집 지점 필요 여부 확인

## 오픈 질문

- 자동 삭제 시간 변경을 설정 파일로 관리할지 상수로 관리할지
- 자동 삭제 완료 후 사용자에게 별도 알림이 필요한지
