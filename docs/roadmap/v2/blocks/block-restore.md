# Block Restore v2 Roadmap

## 현재 기준

- v1에서는 block restore server API를 제공하지 않는다.
- 같은 브라우저 세션 안의 직전 편집 복구는 클라이언트 undo/redo가 담당한다.
- 영속 복구는 문서 soft delete 복구 API만 서버가 담당한다.

## 재검토 조건

- 세션 종료 후에도 블록 단위 복구가 필요하다는 사용자 요구가 확인될 때
- 다른 사용자나 다른 기기에서 수행한 삭제를 부분 복구해야 할 때
- 블록 휴지통, 감사 로그 기반 복구, 운영자 수동 복원이 필요해질 때

## 후속 Todo

- 블록 단위 복구가 필요한 실제 사용자 시나리오 수집
- 부분 복구 시 부모 블록 상태 검증 규칙 정의
- 원래 위치 복구 실패 시 대체 배치 정책 검토
- 블록 휴지통과 문서 복구 정책 경계 정리

## 검증 보강

- 대형 블록 트리에서 partial restore 범위 계산 비용 측정
- 이동 후 삭제된 블록을 복구할 때 정렬 키 충돌 검증
- 협업 편집 중 부분 복구와 optimistic lock 충돌 시나리오 점검

## 정책 / 운영 고려사항

- block restore를 end-user API로 둘지 운영자 도구로 둘지 결정
- 감사 로그 최소 기록 단위를 block action까지 확장할지 결정
- 복구 가능 기간과 purge 정책을 문서 복구와 동일하게 맞출지 검토

## 오픈 질문

- 부분 복구 단위를 단일 블록으로 제한할지 하위 트리 전체로 볼지
- 부모가 삭제 상태거나 이동된 경우 실패 처리와 대체 위치 복구 중 무엇을 기본으로 할지
- restore 이후 version 처리와 이벤트 발행 방식을 어떻게 맞출지
