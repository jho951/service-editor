# Editor Transaction Logging v2 Roadmap

## 현재 기준

- v1 에디터 저장은 `POST /v1/editor-operations/documents/{documentId}/save` 한 경로를 표준으로 사용한다.
- v1 priority는 안정적인 autosave, 단순한 충돌 처리, 예측 가능한 UX다.
- v1 충돌 처리는 block version 기반 optimistic lock과 전체 rollback을 우선한다.
- v1에서는 저장 자체를 먼저 안정화하고, 별도 transaction 로그 테이블은 당장 도입하지 않는다.

## 왜 v2 후보로 두는가

- transaction 로그 테이블은 디버깅, 장애 분석, 운영 추적, 향후 이벤트 기반 확장에 도움이 될 수 있다.
- 반면 모든 editor transaction을 그대로 영구 적재하면 저장량, 쓰기 부하, 보관 정책 비용이 빠르게 커질 수 있다.
- 특히 autosave 빈도가 높은 문서에서는 "실용성 없는 중간 저장 로그"가 과도하게 쌓일 위험이 있다.

따라서 이 기능은 "있으면 좋아 보이는 기능"이 아니라, 실제 운영 문제를 해결할 만큼 실용성이 있는지 확인한 뒤 제한적으로 도입해야 한다.

## 후속 Todo

- transaction 로그 테이블이 실제로 필요한 운영 질문을 먼저 정의
- 예: 저장 실패 추적, 충돌 재현, 특정 문서의 비정상 편집 흐름 분석
- 전체 payload를 저장할지, 최소 메타데이터만 저장할지 비교
- 보관 기간과 purge 정책이 없으면 도입하지 않는다
- 개인정보/민감 본문을 로그에 그대로 남겨도 되는지 검토

## 검증 보강

- autosave 빈도 기준 예상 로그 적재량 측정
- 문서 크기와 사용 패턴별 저장 비용 추정
- 장애 재현 시 실제로 어떤 필드가 있어야 도움이 되는지 운영 관점 점검

## 정책 / 운영 고려사항

- 기본 원칙:
- 디버깅에 실제로 필요한 최소 정보만 남긴다.
- 본문 전체 structured content는 기본 저장 대상이 아니다.
- 문서/사용자/배치 식별자, 처리 결과, 충돌 여부, 처리 시간 같은 메타데이터 우선

- 예외 원칙:
- 특정 장애 분석이 필요할 때만 샘플링 또는 단기 보관 전략 검토

- 제외 후보:
- 모든 transaction payload 영구 적재
- 모든 autosave content snapshot 영구 적재
- 조회되지도 않는 과도한 세부 로깅

## 선행조건 / 의존성

- v1 transaction API 안정화
- 운영 환경에서 저장 실패/충돌/재시도 패턴 관측
- 실제 장애 대응 요구 수집

## 오픈 질문

- 최소 로그 필드는 어디까지인가
- payload를 저장하지 않고도 운영 문제를 해결할 수 있는가
- 샘플링, TTL, 압축 없이도 유지 가능한가
- 로그 테이블보다 애플리케이션 메트릭/structured log만으로 충분한가
