# Block Delete v2 Roadmap

## 현재 기준

- v1 블록 삭제는 soft delete 기반이다.
- 삭제 대상 루트 블록과 그 하위 활성 블록 전체를 함께 soft delete 한다.
- `deletedAt`은 동일한 시각으로 기록하고, `updatedBy`는 요청 사용자 기준으로 갱신한다.
- 실제 DB 반영은 수집된 블록 ID 목록에 대해 bulk update 1회로 처리한다.

## 후속 Todo

- 삭제 응답에 삭제된 블록 수 또는 영향 범위를 노출할 필요가 있는지 검토
- 블록 삭제와 문서 삭제 경로의 공통 처리 규칙 정리
- 삭제 이벤트 또는 감사 로그 적재 필요 여부 검토
- 대형 트리 삭제 시 사용자 피드백 방식 검토

## 검증 보강

- 깊은 트리 구조에서 삭제 성능과 쿼리 횟수 측정
- sibling이 많은 구조에서 삭제 대상 외 블록 보존 검증 강화
- 이미 soft delete된 하위 블록이 섞여 있을 때 동작 검증
- 이동 요청 또는 수정 요청과 삭제 요청이 겹칠 때의 충돌 시나리오 점검

## 정책 / 운영 고려사항

- 하위 블록까지 함께 삭제된다는 사실을 API 또는 클라이언트에서 어떻게 명시할지 정리
- 대형 삭제 요청은 동기 처리로 유지할지 비동기 전환이 필요한지 여부
- 운영 로그와 감사 로그의 최소 기록 단위 정리

## 선행조건 / 의존성

- 운영 환경 기준 대형 문서/대형 블록 트리 규모 파악
- 감사 로그 또는 이벤트 발행 요구사항 존재 여부 확인

확정된 전제:

- v1에서는 block restore server API를 두지 않는다.
- 같은 세션의 직전 삭제 복구는 클라이언트 undo/redo가 담당한다.

## 오픈 질문

- 삭제 대상 중 일부가 이미 soft delete 상태여도 성공 응답으로 유지할 것인가
- block delete와 document delete의 감사 추적 포맷을 통일할 것인가
- 대형 트리 삭제 시 응답 시간 목표를 별도로 둘 것인가
