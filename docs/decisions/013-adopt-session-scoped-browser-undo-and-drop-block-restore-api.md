# ADR 013: 브라우저 세션 undo/redo를 채택하고 v1 블록 복구 API는 제외한다

## 상태

채택됨

## 배경

v1 편집기에서는 사용자가 방금 수행한 블록 수정, 삭제, 이동을 빠르게 되돌릴 수 있어야 한다.

하지만 이 요구를 server restore API로 풀면 다음 문제가 생긴다.

- 블록 단위 복구는 부모 블록 상태, 원래 위치, sibling 재배치 규칙까지 함께 정의해야 한다.
- 문서 복구 API가 이미 존재하는 상황에서 블록 복구 API까지 두면 영속 복구 경계가 중첩된다.
- 현재 팀은 undo/redo를 브라우저 세션 범위의 로컬 캐시 기반 기능으로 운영하고, 탭 또는 브라우저 종료 후 복구는 지원하지 않는 방향을 선택했다.

외부 유사 사례도 편집 중 undo/redo와 문서/페이지 단위 영속 복구를 분리하는 경향이 강하다.

## 결정

- v1의 블록 단위 직전 편집 복구는 브라우저 세션 범위의 undo/redo로 처리한다.
- 탭 또는 브라우저 종료 후에는 블록 단위 undo/redo를 보장하지 않는다.
- server 측 영속 복구 API는 문서 soft delete 복구만 제공한다.
- `POST /v1/blocks/{blockId}/restore`는 v1 요구사항과 API 명세에서 제외한다.
- 문서 복구 시에는 기존 정책대로 해당 문서와 하위 문서, 소속 블록을 함께 복구한다.
- 세션 종료 후 블록 단위 복구, 블록 휴지통, 감사 로그 기반 부분 복원은 v2 이후 별도 정책으로 재검토한다.

## 영향

- 장점:
- v1 서버 범위가 명확해지고 구현 복잡도가 줄어든다.
- 사용자가 기대하는 즉시 되돌리기 경험을 클라이언트에서 더 빠르게 제공할 수 있다.
- 문서 복구와 편집 undo의 책임 경계가 분리된다.

- 단점:
- 브라우저 세션 종료 후에는 블록 단위 부분 복구를 제공하지 못한다.
- 다른 기기나 다른 사용자가 수행한 블록 삭제를 부분적으로 되돌리는 시나리오는 지원하지 않는다.
- 향후 부분 복구 요구가 커지면 정책과 API를 다시 설계해야 한다.
