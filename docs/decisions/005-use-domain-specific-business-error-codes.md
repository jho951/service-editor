# ADR 005: Business 오류 코드는 도메인별로 명시한다

## 상태

채택됨

## 배경

현재 비즈니스 예외는 `RESOURCE_NOT_FOUND`처럼 범용 이름으로도 표현할 수 있다.
하지만 Document, Workspace, Block처럼 도메인이 늘어나면 같은 `404`라도 어떤 리소스가 실패했는지 응답 코드와 메시지만으로 구분하기 어려워진다.
이 경우 클라이언트 분기, 운영 로그 분석, 테스트 가독성이 모두 떨어진다.

## 결정

- `BusinessErrorCode`와 API `ErrorCode`는 enum 이름 일치 규칙을 유지한다.
- `NOT_FOUND` 계열은 범용 `RESOURCE_NOT_FOUND`를 기본으로 쓰지 않고, `WORKSPACE_NOT_FOUND`, `DOCUMENT_NOT_FOUND`, `BLOCK_NOT_FOUND`처럼 도메인별 코드로 정의한다.
- 같은 원칙을 향후 `CONFLICT`, `FORBIDDEN` 등 도메인 맥락이 필요한 오류에도 우선 적용한다.
- 메시지는 어떤 도메인 리소스가 실패했는지 직접 드러내는 문장으로 작성한다.

## 영향

- 장점:
  - 클라이언트가 도메인별 실패를 더 정확하게 처리할 수 있다.
  - 테스트와 운영 로그에서 실패 원인이 더 명확해진다.
  - 이후 Block 등 후속 도메인 확장 시 오류 코드 체계를 일관되게 늘릴 수 있다.
- 단점:
  - 코드 수가 늘어나므로 enum과 문서 관리 비용이 조금 증가한다.
  - 너무 세분화하면 실제로 구분 가치가 낮은 코드까지 늘어날 수 있어 명명 기준을 유지해야 한다.
