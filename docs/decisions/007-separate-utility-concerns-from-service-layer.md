# ADR 007: 서비스 계층에서 유틸성 파싱/정규화 책임을 분리한다

## 상태

채택됨

## 배경

현재 `DocumentServiceImpl`, `WorkspaceServiceImpl`는 문서/워크스페이스 생성 유스케이스를 처리하면서 문자열 `trim`, nullable actor 정규화, `sortKey` 파싱/포맷팅까지 직접 수행하고 있었다.
이 구조는 서비스가 비즈니스 규칙 검증과 기술성 유틸 로직을 함께 품게 만들어 책임 경계를 흐리고, 재사용 가능한 로직의 테스트 위치도 애매하게 만든다.

또한 API 계층의 `DocumentApiMapper`는 응답/요청 필드 매핑 외에 JSON 직렬화와 역직렬화까지 포함하고 있어, mapper가 단순 변환기 이상의 책임을 가지는 상태였다.

## 결정

- Service 계층은 유스케이스 오케스트레이션, 트랜잭션 경계, 도메인 규칙 검증만 담당한다.
- 문자열 정규화, `sortKey` 계산 같은 재사용 가능한 기술성 처리 로직은 현재 모듈 구조 안에서 별도 유틸/지원 클래스(`support`)로 분리한다.
- 문서/블록 `sortKey` 계산은 재사용 가능한 `OrderedSortKeyGenerator` 지원 객체로 분리한다.
- API 계층의 JSON 직렬화/역직렬화는 mapper 외부의 codec 지원 객체로 분리한다.
- 포트/어댑터 추상화는 실제 교체 가능성이나 외부 경계가 분명해질 때만 도입하고, 현재는 단순한 계층 구조를 유지한다.

## 영향

- 장점:
  - 서비스 코드가 비즈니스 로직 중심으로 단순해진다.
  - 정규화/파싱/포맷팅 규칙을 독립 테스트 대상으로 다룰 수 있다.
  - 새 유스케이스가 추가돼도 같은 지원 컴포넌트를 재사용하기 쉬워진다.
  - 현재 코드베이스의 단순한 계층 구조와 충돌하지 않는다.
- 단점:
  - 향후 교체 가능한 외부 경계가 생기면 그 시점에 추상화를 다시 도입해야 할 수 있다.
  - 유틸 클래스가 무분별하게 늘어나지 않도록 범위를 관리해야 한다.
