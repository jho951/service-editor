# 2026-03-25 문서 hard delete 및 PATCH 휴지통 엔드포인트 정책 검토 메모

## 문서 목적

- 기존 문서 삭제 API를 hard delete로 전환할지 검토한다.
- 휴지통 기능을 문서 기본 삭제 API와 분리해 별도 엔드포인트로 둘지 검토한다.
- 현재 테스트 기준 자동 삭제 시간 5분을 휴지통 정책에 어떻게 연결할지 결정 근거를 남긴다.
- 이 문서는 전략 비교와 정책 정리를 위한 검토 메모이며, 실제 채택안은 다음 단계에서 요구사항 문서와 구현으로 반영한다.

## 배경

- 현재 구조는 문서 삭제를 `deletedAt` 기반 soft delete로 처리하고, 복구도 같은 문서 API 흐름 안에서 다룬다.
- 이번 변경 방향은 "문서 삭제"와 "휴지통으로 보내기"를 서로 다른 액션으로 분리하는 것이다.
- 이 기준에서는 사용자가 문서를 삭제하면 즉시 물리 삭제되고, 휴지통 보관이 필요한 경우에는 별도 휴지통 엔드포인트를 사용해야 한다.
- 휴지통은 문서 목록/단건 조회와 다른 수명 주기를 가지므로, 기본 문서 API와 분리하면 의미가 더 명확해진다.
- 현재 단계에서는 테스트와 검증을 위해 휴지통 자동 영구 삭제 시간을 우선 5분으로 두고, 실제 운영 정책은 추후 조정 가능하도록 열어둔다.

## 검토 범위

- 문서 기본 삭제 API를 hard delete로 둘지
- 휴지통 진입을 별도 엔드포인트로 분리할지
- 휴지통 자동 영구 삭제 기준 시각과 실행 주기
- 휴지통 복구 허용 여부와 실패 조건
- 하위 문서와 소속 블록의 hard delete / 휴지통 처리 범위

이번 문서에서 다루지 않는 항목:

- 권한 정책 변경
- 운영자 전용 복구 도구
- 장기 보관형 감사 로그 저장소 상세 설계
- 블록 단위 독립 휴지통 정책

## 핵심 질문

1. `DELETE /documents/{documentId}`를 hard delete로 바꾸는 것이 API 의미상 더 명확한가
2. 휴지통은 `PATCH /v1/documents/{documentId}/trash` 같은 별도 상태 변경 엔드포인트로 분리하는 편이 더 적합한가
3. 휴지통 문서는 현재 테스트 기준 `deletedAt + 5분`이 지나면 자동 영구 삭제되도록 해도 되는가
4. hard delete와 휴지통 진입이 공존할 때 하위 문서와 소속 블록 정리 범위를 어떻게 맞출 것인가
5. 휴지통 자동 삭제는 애플리케이션 스케줄러로 처리하는 편이 현재 구조에 가장 단순한가

## 선택지

### 선택지 1. 기본 삭제는 hard delete, 휴지통은 별도 엔드포인트로 분리

#### 개요

- `DELETE /v1/documents/{documentId}`는 대상 문서, 하위 문서, 각 문서 소속 블록을 즉시 물리 삭제한다.
- 휴지통 보관이 필요한 경우 `PATCH /v1/documents/{documentId}/trash` 같은 별도 엔드포인트를 사용한다.
- 휴지통 진입 시 `deletedAt`을 기록하고, 현재 테스트 기준 `deletedAt + 5분`이 지나면 자동 영구 삭제한다.
- 휴지통 복구는 별도 복구 엔드포인트로 수행한다.

#### 시나리오

1. 사용자가 문서를 완전히 삭제하려고 할 때 `DELETE /v1/documents/{documentId}`를 호출한다.
2. 서버는 대상 문서, 하위 문서, 각 문서 소속 블록을 즉시 물리 삭제한다.
3. 사용자가 문서를 잠시 보관하고 싶을 때 `PATCH /v1/documents/{documentId}/trash`를 호출한다.
4. 서버는 대상 문서 트리와 소속 블록을 휴지통 상태로 전환하고 `deletedAt`을 기록한다.
5. 사용자가 현재 테스트 기준 5분 안에 복구를 요청하면 서버는 부모 상태와 만료 여부를 확인한 뒤 복구한다.
6. 스케줄러는 만료된 휴지통 문서를 찾아 최종 영구 삭제한다.

#### 장점

- 기본 삭제와 휴지통 보관의 의미가 명확하게 분리된다.
- hard delete가 필요한 경로와 복구 가능한 경로를 사용자가 명확히 구분할 수 있다.
- 휴지통 정책이 문서 기본 CRUD 의미를 덜 오염시킨다.

#### 단점

- 사용자와 클라이언트는 삭제 의도를 두 경로 중 하나로 명시적으로 선택해야 한다.
- API 수가 늘고, UX와 연동 정책 설명이 추가로 필요하다.
- 현재 구현은 soft delete 중심이므로 코드 변경 폭이 커진다.

#### 트레이드오프

- API 의미를 명확히 얻는 대신, 현재 구조 재사용성은 줄어든다.

#### 적합한 상황

- "즉시 삭제"와 "복구 가능한 휴지통 이동"을 제품적으로 명확히 분리하고 싶은 상황

### 선택지 2. 기본 삭제는 휴지통 진입으로 유지

#### 개요

- 기존처럼 문서 삭제 자체를 휴지통 진입으로 해석한다.
- 별도 휴지통 엔드포인트는 두지 않는다.

#### 장점

- 현재 구현 재사용성이 높다.
- API 추가가 적다.

#### 단점

- 사용자가 "완전 삭제"와 "휴지통 이동"을 구분할 수 없다.
- 기본 삭제 API 의미가 soft delete에 묶인다.

#### 트레이드오프

- 구현 단순성을 얻는 대신, 삭제 의도 표현은 약해진다.

#### 적합한 상황

- 복구 가능한 삭제 하나만 지원하면 되는 상황

## 비교 요약

- 선택지 1은 API 의미가 더 명확하고, hard delete와 휴지통을 서로 다른 액션으로 구분할 수 있다.
- 선택지 2는 구현은 간단하지만, 현재 변경 방향과 맞지 않는다.
- 이번 변경 방향에서는 선택지 1이 요구 의도와 일치한다.

## 추천 시나리오

- 사용자가 "완전히 삭제"를 선택하면 문서는 즉시 사라지고 복구할 수 없다.
- 사용자가 "휴지통으로 보내기"를 선택하면 문서는 일반 문서 목록에서 사라지지만, 현재 테스트 기준 5분 안에는 복구할 수 있다.
- 사용자가 복구하지 않으면 스케줄러가 만료된 휴지통 문서를 자동 영구 삭제한다.

## 현재 추천 방향

- 기본 문서 삭제 API는 hard delete로 전환한다.
- 휴지통 진입은 별도 엔드포인트로 분리한다.
- 휴지통 상태는 `deletedAt` 기반 soft delete로 표현한다.
- 휴지통 자동 영구 삭제 기준은 현재 테스트용으로 `deletedAt + 5분`으로 둔다.
- 자동 영구 삭제는 애플리케이션 스케줄러 기반으로 처리한다.
- 휴지통 복구는 허용하되, 현재 테스트 기준으로 `deletedAt + 5분`이 지나지 않은 문서만 대상으로 본다.
- hard delete와 휴지통 자동 영구 삭제 모두 대상 문서, 하위 문서, 각 문서 소속 블록까지 같은 범위로 처리한다.
- 자동 삭제 시간은 상수 또는 설정값으로 분리해 추후 정책 변경 시 문서와 구현을 함께 조정할 수 있게 한다.

추천 이유:

- 삭제와 보관의 의미를 분리하면 API 책임 경계가 선명하다.
- "완전 삭제"와 "복구 가능한 삭제"를 동시에 지원할 수 있다.
- 휴지통 정책은 유지하되, 일반 delete API는 hard delete 의미에 집중할 수 있다.

## 수동 복구 허용 여부와 실패 조건

- 수동 복구는 휴지통 엔드포인트로 들어간 문서에 대해서만 허용한다.
- 복구 성공 조건:
  1. 대상 문서가 휴지통 상태여야 한다.
  2. 현재 시각이 `deletedAt + 현재 자동 삭제 시간` 이전이어야 한다.
  3. 부모 문서가 있는 경우, 부모 문서가 활성 상태여야 한다.
- 복구 실패 조건:
  1. 대상 문서가 존재하지 않음
  2. 대상 문서가 이미 활성 상태임
  3. `deletedAt + 현재 자동 삭제 시간`이 지남
  4. 부모 문서가 여전히 삭제 상태임
  5. 자동 영구 삭제가 먼저 완료되어 물리적으로 삭제됨

## 스케줄러 주기와 지연 허용 범위

- 스케줄러는 1분 주기를 기본 추천안으로 둔다.
- 현재 테스트 기준 자동 영구 삭제 대상 조건은 `deletedAt <= now - 5분`이다.
- 실제 삭제 시점은 "정확히 5분 0초"가 아니라 "5분 경과 후 다음 스케줄 실행 시점"이 된다.
- 운영 정책 문구는 "현재 테스트 기준으로 휴지통 이동 후 5분이 지나면 자동 영구 삭제 대상이 되며, 최대 1분 이내 지연될 수 있다"로 표현하는 것이 구현과 맞다.

## 문서 / 하위 문서 / 블록 처리 범위

- hard delete 범위는 대상 문서 1건으로 제한하지 않는다.
- 대상 문서의 하위 문서 전체를 함께 hard delete 한다.
- 대상 문서와 하위 문서 각각에 소속된 블록 전체를 함께 hard delete 한다.
- 휴지통 진입도 동일 범위로 처리해 문서, 하위 문서, 블록 생명주기를 맞춘다.

## 동시성 충돌과 예외 상황 처리

1. hard delete와 휴지통 진입이 동시에 경합하는 경우
   - 먼저 커밋된 쪽이 최종 상태를 결정한다.
   - hard delete가 먼저 끝나면 이후 휴지통 진입은 문서 없음으로 실패해야 한다.

2. 복구와 자동 영구 삭제가 동시에 경합하는 경우
   - 자동 영구 삭제 전에 복구가 먼저 커밋되면 대상은 더 이상 휴지통 조건을 만족하지 않아야 한다.
   - 자동 영구 삭제가 먼저 끝나면 이후 복구는 문서 없음으로 실패해야 한다.

3. 부모 삭제 상태에서 자식만 복구하려는 경우
   - 실패 처리해야 한다.

4. 스케줄러 실행 중 일부 대상 처리 실패
   - 한 문서 트리 단위는 하나의 트랜잭션으로 묶어 일부만 삭제되고 끝나지 않게 해야 한다.
   - 실패한 대상은 다음 스케줄에서 재시도 가능해야 한다.

## 운영 시나리오 예시

### 시나리오 1. 기본 삭제로 즉시 영구 삭제

1. 사용자가 문서를 완전히 삭제한다.
2. 서버는 대상 문서, 하위 문서, 소속 블록을 즉시 물리 삭제한다.
3. 이후 해당 문서는 복구할 수 없다.

### 시나리오 2. 휴지통 이동 후 2분 내 복구 성공

1. 사용자가 문서를 휴지통으로 보낸다.
2. 서버는 문서, 하위 문서, 소속 블록에 `deletedAt`을 기록한다.
3. 사용자가 2분 뒤 복구를 요청한다.
4. 서버는 아직 만료 전이므로 복구를 수행한다.

### 시나리오 3. 현재 테스트 기준 5분 경과 후 자동 영구 삭제

1. 사용자가 문서를 휴지통으로 보낸다.
2. 문서는 일반 문서 목록에서 제외된다.
3. 스케줄러가 만료 시점 이후 실행되면 해당 문서 트리를 물리 삭제한다.

## 미해결 쟁점

1. 휴지통 엔드포인트 경로를 `PATCH /v1/documents/{documentId}/trash`로 둘지 다른 상태 변경 경로로 둘지
2. hard delete를 end-user 기능으로 그대로 노출할지 운영/관리 경로로 제한할지
3. 자동 영구 삭제 시간 변경을 설정 파일로 관리할지 상수로 관리할지
4. 자동 영구 삭제 실행 로그와 모니터링 포인트를 어디까지 남길지
