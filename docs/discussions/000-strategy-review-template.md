# YYYY-MM-DD 주제명 검토 메모

## 문서 목적

- 이 문서에서 무엇을 검토하는지 적는다.
- 왜 지금 이 논의가 필요한지 적는다.
- 문서가 초안인지, 회의록인지, 전략 비교 메모인지 적는다.

## 배경

- 현재 시스템 전제
- 관련 요구사항과 제약
- 이 문서를 읽기 전에 알아야 할 최소 맥락
- 필요하면 배경 말미에 `참고한 자료와 사례` 하위 구간을 둔다.
- 자료 목록은 맥락 이해를 돕는 정도로만 적고, 문서 흐름을 끊을 만큼 길게 늘어놓지 않는다.
- 외부 공식 문서나 외부 사례를 참고했다면, 뒤 본문에서도 `공식 문서 기준`, `공개 자료 기준`, `현재 구조에 대입하면`처럼 출처 성격과 해석을 구분해 적는다.

## 검토 범위

- 이번 문서에서 다룰 항목
- 이번 문서에서 다루지 않을 항목

## 핵심 질문

1. 이번 검토에서 답해야 하는 질문을 적는다.
2. 선택해야 하는 쟁점을 적는다.

## 선택지

작성 규칙:

- 각 선택지는 `개요 -> 시나리오 -> 장점 -> 단점 -> 트레이드오프 -> 적합한 상황` 흐름을 유지한다.
- 한 선택지 안에서 나온 질문은 가능하면 그 선택지 안에서 바로 설명과 판단까지 이어서 끝낸다.
- 다른 선택지에서도 반복되는 공통 근거는 여기서 여러 번 장황하게 반복하지 말고, 아래 `비교 요약`에서 한 번 더 묶어 정리한다.

## 비교 요약

- 선택지 간 차이를 짧게 정리한다.
- 구현 난이도, 운영 난이도, 확장성, 실무성 같은 기준으로 요약한다.
- 각 선택지에서 반복된 공통 논점을 여기서 한 번만 묶어 정리한다.
- 앞에서 나온 비교 포인트를 새로 흩뜨리지 말고, 추천 방향으로 넘어가기 위한 정리 구간으로 사용한다.

## 현재 추천 방향

- 현 시점의 1순위 추천안을 적는다.
- 추천 이유를 3~5개 정도 적는다.
- 가능하면 `추천안 -> 왜 이 안이 현재 맞는가 -> 다른 안은 왜 지금 아닌가` 순서로 적는다.
- 앞에서 제기한 핵심 질문이 이 섹션에서 자연스럽게 회수되도록 작성한다.

## 추천 시나리오

- 왜 이 추천안이 실제 사용 흐름에서 이해하기 쉬운지 적는다.
- 처음 보는 사람도 머릿속에 그릴 수 있도록 대표 시나리오를 짧게 적는다.

## 미해결 쟁점

1. 아직 팀이 결정하지 않은 문제를 적는다.
2. 추가 확인이 필요한 정책을 적는다.

- 작업 로그 링크

## 작성 규칙

- 전략 설명이나 선택지 비교에는 시나리오를 반드시 포함한다.
- 처음 보는 사람이 한 번에 이해할 수 있도록 추상 설명보다 흐름 설명을 우선한다.
- review 문서와 strategy 문서는 이름만 다를 수 있고, 구조는 같은 템플릿 계열로 맞춘다.
- 문서는 섹션 단위로 끊긴 정보 묶음이 아니라, 처음부터 끝까지 읽을 때 판단이 자연스럽게 이어지는 흐름을 목표로 한다.
- 같은 근거와 같은 결론을 문서 여러 위치에서 반복하지 않는다.
- 앞 섹션에서 제기한 문제는 가능한 한 다음 섹션에서 바로 회수하고, unrelated한 새 논점으로 갑자기 점프하지 않는다.
- 참고 자료는 가능하면 배경의 맥락 안에서 소화하고, 별도 목록이 문서 흐름을 끊지 않게 한다.
- 외부 자료를 인용할 때는 사실과 현재 문서의 해석을 구분해 적는다.
