# 트러블슈팅 문서 운영 메모

이 폴더는 프로젝트를 진행하면서 실제로 부딪힌 문제를 "어떻게 진단했고, 어떤 선택지를 비교했고, 왜 그 해법을 골랐으며, 무엇이 달라졌는가"까지 남기는 개인 트러블슈팅 노트 영역이다.

이 문서는 단순 장애 일지를 적기 위한 것이 아니다. 한 번의 문제 해결 과정을 나중에 다시 읽었을 때 아래가 보이도록 정리하는 데 목적이 있다.

- 문제를 어떤 맥락에서 인지했는가
- 원인을 어떻게 좁혀 갔는가
- 해결 후보를 어떤 기준으로 비교했는가
- 왜 특정 방식을 채택했는가
- 적용 후 어떤 결과를 얻었는가
- 가능하면 무엇을 수치로 검증할 수 있는가

중요:

- 실제로 트러블슈팅이 있었던 작업이면 이 폴더 문서를 반드시 생성하거나 기존 문서를 보강한다.
- 문제를 해결했는데 이 폴더에 기록이 없다면, 개인 기준으로는 문제 해결 과정이 충분히 남지 않은 것으로 본다.
- 같은 문제 축의 기록은 반드시 하나의 문서로 유지한다.
- 문제를 처음 인지한 시점, 재현, 가설, 비교, 채택, 적용, 결과 검증은 모두 같은 문서 안에 이어서 누적한다.
- 같은 문제의 후속 실험이나 재검증 때문에 새 파일을 만들지 않는다. 새 파일은 문제 축이 실제로 달라졌을 때만 만든다.

## 언제 반드시 문서화해야 하는가

아래에 해당하면 원칙적으로 트러블슈팅 문서를 남긴다.

- 오류나 장애의 원인을 추적해서 해결한 경우
- 성능 저하, 병목, 과도한 쿼리, 느린 응답을 분석하고 개선한 경우
- 동시성 충돌, stale write, race condition, 정합성 문제를 분석하고 해결한 경우
- flaky test, 환경 의존 실패, 간헐적 부팅 실패처럼 재현 조건을 좁혀 해결한 경우
- 인프라/배포/설정 문제를 분석해 해결한 경우
- 단순 수정이 아니라 해결 후보를 비교하고 하나를 채택한 경우
- 설계나 구현 구조를 바꿔 기존 문제를 해소한 경우

반대로 아래는 보통 의무 대상이 아니다.

- 단순 개념 설명 요청
- 사소한 문구 수정
- 원인 분석이 필요 없는 기계적 rename
- 비교나 판단 없이 끝나는 단순 포맷 정리

판단이 애매하면 남기는 쪽을 기본값으로 잡는다.

## 이 폴더가 다루는 문서의 범위

이 폴더에는 아래처럼 "문제 해결 과정 자체"가 핵심인 문서를 둔다.

- 성능 저하 원인 추적
- 동시성 충돌 원인 분석
- API 오류/예외 흐름 정리
- 인프라/배포/환경 설정 이슈 원인 분석
- 테스트 불안정성(flaky) 원인 분석
- 설계 변경 전후의 문제 해결 판단 기록

반대로 아래는 이 폴더의 주 역할이 아니다.

- 단순 개념 설명
- 팀 공식 결정 문서
- 구현 체크리스트만 있는 가이드
- 회의 메모나 전략 검토 원문

즉:

- 개념을 이해하는 문서면 `learn` 일반 문서
- 팀 선택을 검토하는 문서면 `docs/discussions/`
- 채택된 정책이면 `docs/decisions/`
- 문제 해결의 사고 흐름과 근거를 남기는 문서면 `docs/learn/troubleshooting/`

## 파일명 규칙

- 날짜보다 주제 중심의 고정 파일명을 우선한다.
- 좋은 예:
  - `document-version-conflict-troubleshooting.md`
  - `gateway-header-auth-troubleshooting.md`
  - `docker-local-mysql-boot-failure-troubleshooting.md`
- 피할 예:
  - `2026-03-31-debug.md`
  - `issue-fixed.md`

같은 문제 축의 후속 분석이면 기존 문서를 이어서 보강한다.

## 문서 누적 방식

트러블슈팅 문서는 "한 번 작성하고 끝"이 아니라, 같은 문제 축이 이어지는 동안 계속 보강하는 문서다.

권장 방식:

- 문제를 처음 인지했을 때 문서를 만든다
- 재현 조건이 구체화되면 같은 문서에 추가한다
- 틀린 가설과 버린 이유도 같은 문서에 남긴다
- 해결 후보를 비교하면 같은 문서에 이어 적는다
- 실제 적용 내용과 결과 검증도 같은 문서에 누적한다
- 나중에 수치를 다시 재거나 운영 결과를 확인해도 같은 문서에 보강한다

즉, 문서 하나가 "문제 발생부터 결과까지"의 전체 흐름을 가져야 한다.

## 트러블슈팅 문서에서 가장 중요한 요소

좋은 트러블슈팅 문서는 단순히 "이렇게 고쳤다"로 끝나지 않는다. 최소한 아래 질문에 답해야 한다.

1. 어떤 문제가 있었는가
2. 그 문제가 왜 문제였는가
3. 언제, 어디서, 어떤 조건에서 재현되는가
4. 처음에 어떤 가설들을 세웠는가
5. 어떤 방식으로 가설을 줄였는가
6. 해결 후보는 무엇이 있었는가
7. 각 후보의 장점과 단점은 무엇이었는가
8. 어떤 기준으로 특정 방식을 채택했는가
9. 적용 후 무엇이 달라졌는가
10. 그 결과를 어떻게 검증했는가
11. 아직 남은 리스크나 열어둘 질문은 무엇인가

## 권장 문서 흐름

트러블슈팅 문서는 가능하면 아래 순서를 따른다.

1. `문제 요약`
2. `문제 발생 맥락`
3. `증상과 영향 범위`
4. `재현 조건`
5. `초기 가설`
6. `원인 분석 과정`
7. `해결 방안 후보 비교`
8. `채택한 방식과 이유`
9. `적용 내용`
10. `결과와 검증`
11. `수치화 가능 지표`
12. `남은 리스크와 후속 질문`
13. `최종 요약 블록`

이 순서가 좋은 이유는 독자가 "문제를 봄 -> 왜 그 가설이 나왔는지 이해함 -> 왜 다른 안을 버렸는지 이해함 -> 왜 이 안을 채택했는지 납득함 -> 결과를 확인함" 순서로 읽을 수 있기 때문이다.

## 섹션별로 무엇을 써야 하는가

### 1. 문제 요약

한두 문장으로 끝낸다.

- 무엇이 깨졌는가
- 누가 영향을 받는가
- 어느 흐름에서 드러나는가

나쁜 예:

- "버그 수정"

좋은 예:

- "동시에 같은 문서를 수정할 때 후행 요청이 stale version을 감지하지 못하고 마지막 저장으로 덮어쓰는 문제가 있었다."

### 2. 문제 발생 맥락

문제를 어디서 발견했는지 적는다.

- 로컬 재현
- 통합 테스트
- 실제 운영 로그
- 특정 API 호출
- 프론트 편집 흐름

여기서는 "언제 이 문제가 드러났는가"가 중요하다.

### 3. 증상과 영향 범위

증상과 영향 범위를 분리해서 적는다.

- 증상:
  - 어떤 에러가 보였는가
  - 어떤 잘못된 상태가 저장됐는가
  - 어떤 로그/테스트 실패가 있었는가
- 영향 범위:
  - 특정 엔드포인트만인가
  - 특정 도메인만인가
  - 데이터 정합성까지 깨지는가
  - 사용자 경험만 나빠지는가

### 4. 재현 조건

가능하면 가장 재현 가능한 형태로 적는다.

- 요청 순서
- 입력 데이터
- 선행 상태
- 환경 조건
- 테스트 명령
- 필요한 fixture

좋은 문서는 "이 문서만 보고도 다시 문제를 재현할 수 있는가"를 기준으로 본다.

### 5. 초기 가설

처음에 무엇을 의심했는지 적는다.

예:

- 캐시 문제일 수 있음
- JPA dirty checking 타이밍 문제일 수 있음
- optimistic lock 누락일 수 있음
- 컨트롤러 검증 누락일 수 있음

중요한 점:

- 틀린 가설도 남길 가치가 있다
- 어떤 가설을 먼저 버렸는지까지 보여주면 사고 흐름이 보인다

### 6. 원인 분석 과정

이 섹션이 트러블슈팅 문서의 핵심이다.

여기에는 아래 같은 것을 적는다.

- 어떤 로그를 봤는가
- 어떤 코드를 따라갔는가
- 어떤 쿼리를 확인했는가
- 어떤 테스트를 추가했는가
- 무엇을 확인한 뒤 어떤 가설을 버렸는가

가능하면 "관찰 -> 해석 -> 다음 확인" 순서로 적는다.

예:

- `DocumentTransactionServiceImpl`에서는 version 비교를 하고 있었음
- 하지만 `BlockService.delete(...)`의 bulk update query에는 root version 조건이 없었음
- 즉 사전 검증 직후 다른 요청이 version을 바꿔도 delete는 그대로 통과할 수 있었음

이런 식으로 "어디를 보고 무엇을 확정했는가"가 드러나야 한다.

### 7. 해결 방안 후보 비교

최소 2개 이상 후보를 적는 편이 좋다.

각 후보마다 아래를 적는다.

- 아이디어
- 장점
- 단점
- 도입 비용
- 현재 구조와의 궁합

예:

- 애플리케이션 레벨에서 synchronized 처리
  - 장점: 구현이 단순함
  - 단점: 다중 인스턴스 환경에서 무효
- DB where 절에 version 조건 추가
  - 장점: 현재 optimistic lock 모델과 일관됨
  - 단점: bulk query 반환값 해석이 필요함

### 8. 채택한 방식과 이유

여기서는 "왜 이 방식을 골랐는가"를 분명히 적는다.

좋은 이유 예:

- 현재 도메인 모델과 가장 일관적이었다
- 운영 환경에서 안전했다
- 구현 비용 대비 효과가 컸다
- 테스트로 회귀 방지가 가능했다
- 추후 확장 방향을 막지 않았다

### 9. 적용 내용

실제 무엇을 바꿨는지 적는다.

- 수정한 클래스/계층
- 쿼리 변경
- 테스트 추가
- 문서 반영
- 예외 처리 보강

여기서는 changelog처럼 나열하기보다, "문제 해결에 필요한 변경" 중심으로 적는다.

### 10. 결과와 검증

적용 후 무엇이 달라졌는지 적는다.

- 재현이 사라졌는가
- 테스트가 통과하는가
- 실패 케이스가 기대대로 바뀌는가
- 데이터 정합성이 회복됐는가

검증은 가능하면 아래를 포함한다.

- 실행한 테스트
- 확인한 로그
- 비교한 쿼리 결과
- API 응답 차이

### 11. 수치화 가능 지표

가능하면 이 섹션을 넣는다.

모든 문제가 수치화되지는 않지만, 수치가 가능하면 문서 가치가 크게 올라간다.

예:

- 평균 응답 시간
- p95, p99 지연 시간
- 쿼리 수
- flush 횟수
- 실패 재현 확률
- 충돌 재현 빈도
- heap 사용량
- GC 횟수
- 테스트 실행 시간
- API 성공/실패 비율

중요한 점:

- 실제 수치가 없으면 "무엇을 재면 되는가"라도 적는다
- 측정 방법까지 적으면 나중에 다시 실험하기 쉽다

예:

- "동시 요청 100회 기준 conflict 누락 빈도를 측정한다"
- "JPA SQL 로그와 datasource proxy로 쿼리 수를 비교한다"
- "Boot 통합 테스트의 반복 실행 횟수 대비 flaky 발생률을 기록한다"

### 12. 남은 리스크와 후속 질문

문제를 해결했더라도 끝난 게 아닐 수 있다.

- 다중 인스턴스 환경에서도 같은가
- 운영 트래픽에서 충분히 검증됐는가
- 다른 API에도 같은 패턴이 숨어 있지 않은가
- 더 나은 구조로 일반화할 가치가 있는가

이 질문이 남아 있어야 후속 roadmap이나 discussion으로 연결하기 쉽다.

### 13. 최종 요약 블록

문서 마지막에는 반드시 아래 3개 섹션을 다시 짧게 정리한다.

- `문제 발생`
- `문제 해결 과정`
- `결론`

이 블록은 전체 문서를 다 읽지 않아도 흐름을 빠르게 압축해서 볼 수 있게 해준다.

특히 나중에 발표 자료나 짧은 정리로 옮길 때, 이 마지막 블록이 바로 요약 원본이 되도록 작성한다.

## 권장 작성 톤

- 사실과 관찰을 먼저 적는다.
- 결론만 쓰지 말고 그 결론에 도달한 근거를 적는다.
- 틀렸던 가설도 필요하면 남긴다.
- "잘 안 됐다"보다 "어디를 확인했고 왜 아니라고 판단했는지"를 적는다.
- 추상어보다 구체적인 클래스, 메서드, 테스트, 쿼리, 로그를 적는다.

## 문서가 남기는 인상 기준

이 문서는 나중에 누가 읽더라도 "실제로 문제를 추적하고 해결한 사람이 남긴 기록"처럼 보여야 한다.

따라서 아래 원칙을 지킨다.

- 작업자 관점의 관찰과 판단을 중심으로 쓴다.
- 막연한 미사여구보다 확인한 사실, 버린 가설, 채택 이유를 적는다.
- "AI가 정리해준 요약"처럼 보이는 추상적 문장을 피한다.
- 문제를 실제로 어떻게 좁혀 갔는지 흐름이 보여야 한다.
- 가능하면 테스트, 로그, 쿼리, 코드 경로, 실험 조건처럼 검증 가능한 흔적을 남긴다.

피해야 하는 표현 예:

- "여러 방안을 검토한 결과 최적의 해법을 찾았다"
- "효율적인 방식으로 개선했다"
- "전반적으로 안정성을 높였다"

권장되는 표현 예:

- "`BlockService.delete(...)`의 bulk update query에는 root version 조건이 없어서 사전 검증 뒤 stale delete가 통과할 수 있었다"
- "`DocumentTransactionConcurrencyIntegrationTest`를 추가해 같은 시나리오를 반복 재현했고, delete 이후 update가 deleted row를 되살리는 경우를 확인했다"
- "애플리케이션 레벨 락보다 DB where 절 version 조건 추가가 현재 optimistic lock 모델과 더 일관적이라 이 방식을 채택했다"

즉, 문서는 "문제를 해결했다"보다 "어떻게 확인했고 왜 그렇게 판단했는가"가 읽혀야 한다.

## 수치화가 가능할 때 같이 적으면 좋은 것

### 성능

- 요청 시간 평균, p95, p99
- 특정 API 전/후 응답 시간
- 쿼리 수 전/후 비교
- 캐시 hit ratio

### 동시성

- 동일 시나리오 반복 재현 횟수
- conflict 검출 성공/실패 횟수
- race condition 재현 빈도

### 안정성

- flaky test 재현 비율
- 재시도 전/후 성공률
- 장애 발생 빈도

### 운영/배포

- 부팅 시간
- 이미지 크기
- 배포 시간
- health check 통과 시간

## 측정 방법까지 적는 법

수치만 적지 말고, 어떻게 잴 수 있는지도 같이 적는다.

예:

- `./gradlew test --tests ...` 반복 실행 100회
- `ab`, `wrk`, `k6` 같은 부하 도구 사용
- datasource proxy 또는 SQL 로그로 쿼리 수 비교
- 애플리케이션 로그에서 requestId 기준 흐름 추적
- H2/MySQL에서 실제 row 상태 비교

## 작성 플로우

새 트러블슈팅 문서를 만들 때는 가능하면 아래 순서를 따른다.

1. 문제를 한 문장으로 요약한다.
2. 재현 조건을 먼저 고정한다.
3. 초기 가설을 2개 이상 적는다.
4. 확인한 로그, 코드 경로, 테스트를 기준으로 가설을 줄인다.
5. 해결 후보를 비교한다.
6. 채택 이유를 적는다.
7. 적용 내용을 적는다.
8. 검증 결과를 적는다.
9. 가능하면 수치와 측정 방법을 적는다.
10. 남은 리스크를 적는다.
11. 문서 마지막에 `문제 발생 / 문제 해결 과정 / 결론` 요약 블록을 다시 정리한다.

## 최소 완료 기준

아래가 없으면 트러블슈팅 문서가 충분하지 않은 것으로 본다.

- 문제 요약
- 재현 조건
- 원인 가설 또는 원인 분석 과정
- 해결 후보 비교 또는 최소한 대안 검토 흔적
- 채택 이유
- 결과와 검증
- 마지막 요약 블록

가능하면 아래도 넣는다.

- 수치
- 측정 방법
- 남은 리스크

## 마지막 체크리스트

- 재현 조건이 있는가
- 잘못된 가설과 버린 이유가 보이는가
- 해결 후보 비교가 있는가
- 채택 이유가 명시적인가
- 결과 검증이 있는가
- 가능하면 수치나 측정 방법이 있는가
- 다음에 다시 읽어도 사고 흐름이 보이는가
- 실제로 문제를 추적하고 해결한 사람이 남긴 기록처럼 보이는가
